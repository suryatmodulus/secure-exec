## Create a runtime

`NodeRuntime.create(options?)` returns a ready-to-use runtime. All options are
optional; `cwd` defaults to `/home/user`, and permissions default to a secure
policy that denies network access.

```ts
import { NodeRuntime } from "secure-exec";

// Boot a VM. Every option here is optional; these are the defaults made
// explicit. `cwd` defaults to "/home/user", and network is denied by default.
const rt = await NodeRuntime.create({
  env: { GREETING: "hello from the VM env" },
  cwd: "/home/user",
});
```

*[See Full Example](https://github.com/rivet-dev/secure-exec/tree/main/examples/docs/sdk-overview)*

## Execute code

Two methods run sandboxed code. Guest `code` runs as an ES module, so `import`
and top-level `await` both work.

`exec()` runs code and captures `stdout`, `stderr`, and `exitCode`:

```ts
// exec() runs guest code for its output. Guest code is an ES module, so
// `import` and top-level `await` both work.
const execResult = await rt.exec(
  `
  import os from "node:os";
  console.log("stdout:", 1 + 1);
  console.error("stderr:", process.env.GREETING);
  console.log("platform:", os.platform());
  `,
);
console.log("exec.stdout:  ", JSON.stringify(execResult.stdout));
console.log("exec.stderr:  ", JSON.stringify(execResult.stderr));
console.log("exec.exitCode:", execResult.exitCode);
```

`run()` additionally returns a JSON-serializable `value`. The guest calls
`globalThis.__return(value)` and that value is decoded on the host:

```ts
// run() returns a JSON-serializable value: the guest calls
// globalThis.__return(value) and that value is decoded on the host.
const runResult = await rt.run<{ sum: number; cwd: string }>(
  `
  globalThis.__return({ sum: 2 + 40, cwd: process.cwd() });
  console.log("computed in the VM");
  `,
);
console.log("run.value:    ", JSON.stringify(runResult.value));
console.log("run.stdout:   ", JSON.stringify(runResult.stdout));
console.log("run.exitCode: ", runResult.exitCode);
```

## Per-run options

`exec()` and `run()` accept per-call options: extra `env`, a `cwd`, `stdin` data
(`string` or `Uint8Array`), and a `timeout` in milliseconds.

```ts
// Per-run options: extra env, stdin, cwd, and a timeout (ms).
const stdinResult = await rt.run<string>(
  `
  let input = "";
  for await (const chunk of process.stdin) input += chunk;
  globalThis.__return(input.trim().toUpperCase());
  `,
  { stdin: "piped through stdin", timeout: 10_000 },
);
console.log("run.value (stdin):", JSON.stringify(stdinResult.value));
```

## Permissions

The `permissions` option to `NodeRuntime.create()` controls what guest code can
access. It defaults to a secure policy that denies network access while leaving
the `fs`, `childProcess`, `process`, and `env` scopes enabled. Your policy is
merged over that secure default, so a partial policy works: `{ network: "allow" }`
grants the network while keeping the execution essentials in place. The policy
covers `fs`, `network`, `childProcess`, `process`, `env`, and `tool` scopes, each
set to `"allow"`, `"deny"`, or a rule set.

Because secure-exec runs each program as a guest `node` process, `childProcess`
must remain allowed for `exec()` / `run()` to work. Since the default already
denies network access, the snippet below is equivalent to the default and shown
to make the secure baseline explicit:

```ts
const rt = await NodeRuntime.create({
  permissions: {
    fs: "allow",
    childProcess: "allow",
    network: "deny",
  },
});
```

See [Permissions](/docs/features/permissions) for the full policy shape.

## Clean up

Always release the VM and the underlying sidecar when you are done:

```ts
// Always release the VM and the underlying sidecar.
await rt.dispose();
console.log("disposed");
```
