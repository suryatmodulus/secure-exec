Secure Exec runs untrusted code safely on behalf of a trusted caller. This page is the on-ramp: it introduces the three components of a runtime, the trust boundary that separates them, and what happens when you ask the runtime to execute some code. Once the shape clicks, the [Architecture](/docs/architecture) reference goes deeper into the packages, crates, kernel-owned syscall paths, and the diagram of the trust boundary.

## Three components

Every Secure Exec runtime is built from three roles.

- **Client.** The trusted caller. It speaks the wire protocol and asks the sidecar to create VMs and run code. It never runs guest code itself, and everything it configures (the VM setup, the permission policy, resource limits) is trusted. In native mode this is `NodeRuntime`; there is a matching Rust client.
- **Sidecar.** The trusted enforcement point, and the part of the system that actually owns everything: the kernel, the virtual filesystem, the process and socket tables, pipes, PTYs, the permission policy, and DNS. It creates executors on demand and services every request they make.
- **Executor.** The untrusted leaf. It runs the guest code (your JavaScript, plus any third-party or generated code) and reaches the kernel for every syscall. It holds no capabilities of its own; it can only ask.

The guest never runs in the client. It runs inside the executor, behind the kernel's isolation boundary, and the only way out is through the sidecar.

## The trust boundary

The one line that matters is between the **sidecar** and the **executor**.

Everything the client sends is trusted configuration. The sidecar is trusted too: it is the part of the system responsible for enforcing the boundary. The executor is the adversary. The single thing the client contributes that is **not** trusted is the code or payload it submits, because that code is what runs in the executor.

So when the guest tries to read a file, spawn a process, or open a socket, it cannot do any of those things directly. It makes a request that crosses into the sidecar, where the kernel checks the applied permission policy and then services the request against the VM's virtualized resources. The executor never touches a real filesystem, a real socket, or a real host process.

<Aside type="note">A weakness that only shows up when the client supplies a malicious config or credential is not a sandbox escape: the client is configuring its own VM and already controls the host. The boundary that matters is keeping the untrusted guest inside the kernel. See the [Security Model](/docs/security-model).</Aside>

## Lifecycle of a request

Here is what happens, end to end, when you run a piece of code. Starting a runtime spawns the sidecar, opens a session, and creates a VM with a bootstrapped filesystem; from there each call follows the same path.

1. **You call the client.** For example `rt.exec(code)`. The client prepares the request: it stages the guest code inside the VM and asks the kernel to run it as a fresh process.
2. **The request crosses to the sidecar.** It travels over the wire protocol (hop one) from the client to the sidecar that owns the kernel.
3. **The sidecar starts the guest.** The kernel spawns the process, and the executor begins evaluating the guest code.
4. **The guest makes syscalls.** Reading files, spawning child processes, reading env, and (when permitted) reaching the network are all requests from the executor back into the kernel (hop two). The executor blocks until the kernel replies.
5. **The kernel services each syscall.** It applies the permission policy, then carries out the operation against the VM's virtualized resources. Even `fetch()` flows through the kernel's socket table, so network access stays inside the boundary.
6. **Results travel back.** Captured stdout, stderr, and the exit code cross the wire back to the client and resolve your call.

Two hops, then, in every runtime: **client to sidecar** (a request/response protocol) and **executor to kernel** (a blocking syscall path). The [Architecture](/docs/architecture) reference covers how these hops are carried and how the kernel-owned paths fit together.

## A minimal run

```ts
import { NodeRuntime } from "secure-exec";

const rt = await NodeRuntime.create();
try {
  const { stdout } = await rt.exec("console.log('hi', 1 + 1)");
  console.log(stdout.trim()); // hi 2
} finally {
  await rt.dispose();
}
```

`NodeRuntime.create()` is the client. It spawns the sidecar, creates the VM, and mounts the runtimes that execute guest code. `exec()` runs your code as a guest process behind the boundary, and `dispose()` tears the runtime down.
