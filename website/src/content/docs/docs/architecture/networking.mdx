## One transport: the kernel socket table

The design rule is simple: there is a single authoritative transport for everything VM-local, the kernel socket table. Guest `fetch()`, `node:http`, `node:net`, WASM TCP clients and servers, and host `rt.fetch()` all target the same listener table. The kernel pairs two in-VM sockets together and moves bytes between them. It never opens a real host socket for guest work.

Earlier versions carried two listener models at once:

- **Stream-mode** listeners (`net.createServer`, WASM) used real kernel TCP sockets.
- **Object-mode** HTTP listeners (`http.createServer`) used a separate sidecar table that exchanged JSON request and response objects over stream events, not actual TCP bytes.

The object-mode path worked for host-to-JS HTTP, but another guest process could not reach it reliably: the client expected byte-stream TCP semantics while the server only spoke object-mode dispatch. The current architecture removes that split. HTTP is layered on top of kernel TCP bytes, so every listener (JS `net`, JS `http`, WASM) lives in the one socket table and is reachable identically.

## The layers

A request passes through four layers. Only the top and bottom understand HTTP; the middle two move bytes and enforce policy.

```text
  Layer 4   Guest bridge (untrusted, in V8 isolate)
            node:http / node:net / fetch / undici
            crates/execution/assets/v8-bridge.source.js
                          |  bridge calls
  Layer 3   Sidecar sync-RPC dispatch (trusted)
            net.connect, net.http_request, net.listen, ...
            crates/sidecar/src/service.rs
                          |
  Layer 2   Sidecar execution and enforcement (trusted, TCB)
            listener state, host fetch client, permission checks
            crates/sidecar/src/execution.rs
                          |  owner-checked kernel calls
  Layer 1   Kernel socket table (trusted, TCB floor)
            bind / listen / connect / read / write, loopback routing
            crates/kernel/src/socket_table.rs, kernel.rs
```

### Layer 1: kernel socket table

`crates/kernel/src/kernel.rs` exposes the POSIX-style primitives (`socket_create`, `socket_bind_inet`, `socket_connect_inet_loopback`, `socket_read`, `socket_write`, `poll_targets`). Every call is owner-checked and resource-accounted, and failures return correct POSIX errnos so guest code branches the way it would on Linux.

Loopback routing is the heart of VM-local networking. `socket_connect_inet_loopback` only succeeds against a socket that is actually bound and listening in the same VM's table; otherwise it returns `ECONNREFUSED`. It also runs the resource-limit checks before pairing the two sockets. No host networking is involved at this layer.

### Layer 2: sidecar execution

`crates/sidecar/src/execution.rs` is the enforcement point (the trusted computing base). It has two roles relevant to networking:

- **Listener state.** `build_javascript_socket_path_context` walks every active process and records what is listening on which port, including a map of HTTP loopback targets keyed by `(family, port)`. This is the source of truth a connect consults to learn "port 3000 is an HTTP server owned by process X, server Y."
- **The host fetch client.** When the host calls `rt.fetch()`, the sidecar resolves the target to a VM-owned kernel listener, opens its own kernel socket, connects over loopback, and speaks HTTP/1.1 to the guest server. This is the only HTTP client that lives in the sidecar (the host has no guest isolate to do framing for it).

### Layer 3: sidecar sync-RPC dispatch

`crates/sidecar/src/service.rs` routes the bridge calls guest code makes. The guest-to-guest loopback HTTP path lands here as `net.http_request`. It is the most security-sensitive RPC, so it is guarded in order: the host must be a loopback address, the applied network policy must permit the operation, and the requested `(process_id, server_id)` must match a listener that is currently live. That last check stops a guest from forging a target to reach a process it should not.

### Layer 4: guest bridge

`crates/execution/assets/v8-bridge.source.js` is the Node-compatibility shim running inside the untrusted V8 isolate. It presents `node:http`, `node:net`, `fetch`, and undici to guest code and translates them into Layer 3 bridge calls. `http.createServer()` is implemented on top of `net.Server`: each accepted byte socket is parsed as HTTP and dispatched to the guest's request handler.

## Where HTTP meets TCP

This is the part worth being precise about: there is no single, shared HTTP/TCP translation module. Because the wire between every endpoint is raw TCP bytes through the kernel, HTTP is framed and deframed at each edge that speaks HTTP. The kernel (Layer 1) and the sidecar routing (Layer 2) never parse HTTP.

```text
   HTTP endpoint A                 raw TCP bytes                 HTTP endpoint B
  (encode + decode)  <-->   kernel socket table (Layer 1)  <-->  (encode + decode)
                            sidecar routing    (Layer 2)
                            -- moves bytes, never parses HTTP --
```

There are three independent HTTP codecs, one per kind of endpoint:

| Endpoint | Direction | Lives in | Encode / decode |
| --- | --- | --- | --- |
| Guest HTTP server | bytes to handler and back | `v8-bridge.source.js` (guest) | `parseLoopbackRequestBuffer` (request bytes to object), `serializeLoopbackResponse` (response object to bytes), wired up per accepted socket by `attachHttpServerSocket` |
| Guest HTTP client | `fetch()` / `http.request()` | `v8-bridge.source.js` (guest) | undici over a kernel-backed dispatcher, or the raw serializer `serializeRawHttpRequest` plus `waitForRawHttpResponse` |
| Host fetch client | `rt.fetch()` | `crates/sidecar/src/execution.rs` (sidecar) | `serialize_kernel_http_fetch_request` (request to bytes), `parse_kernel_http_fetch_response` (bytes to response JSON) |

A WASM HTTP server or client does its own framing in guest code (for example reading the request line and writing a response with the standard C socket calls). The kernel does not help it; it is just bytes, the same as for the JS endpoints.

The practical payoff: a client never needs to know whether the server it is talking to is JS, WASM, raw TCP, or HTTP. Every server is a kernel TCP listener, and HTTP is a guest-side concern layered on top.

## Data flows

These map to the cross-runtime test matrix.

**Host to guest (`rt.fetch`).** The sidecar resolves the port to a VM-owned kernel listener, opens a sidecar-owned kernel socket, connects over loopback, serializes the request bytes, drives the target process forward so it can accept and respond, then parses the response bytes back into the host response object. It is fail-closed: no DNS, no external networking, no host-loopback fallback. If no VM-owned listener exists, it returns a missing-listener error.

**Guest to guest.** The client's `net.connect` goes through the sidecar, which returns a loopback HTTP target handle. The guest then sends the request through `net.http_request`, which dispatches into the target process's request handler. Cross-process loopback now passes through the enforcement point instead of taking an in-isolate shortcut.

**Cross-runtime (JS and WASM, either direction).** Client and server connect through a kernel loopback socket pair and exchange raw bytes. JS to WASM, WASM to JS, and WASM to WASM all use the same path; the only difference is which side runs the HTTP codec.

**Guest outbound to host or external.** Connections that do not target a VM-owned listener take the existing external network path: permission checks, DNS pinning, then a real host `TcpStream`. Reaching a host loopback port still requires an explicit `loopbackExemptPorts` entry.

## Trust and permissions

The security boundary is sidecar to executor. Guest code is untrusted; VM configuration (permission policy, allowlist, limits, loopback exemptions) is trusted.

- Every guest connect, listen, read, and write passes through sidecar ownership and kernel owner checks.
- Host `rt.fetch()` to a VM-owned listener is host control-plane traffic, not guest egress. It works when the guest server is allowed to listen even if guest outbound fetch and connect are denied, and it only ever reaches VM-owned listeners.
- Guest-to-guest loopback is allowed only when the destination is a VM-owned listener and the applied network policy permits the connect.
- Host-loopback access from guest code is separate and still requires `loopbackExemptPorts` plus the applied network policy.
- Long-lived waits must not block the sync-RPC path; the stack uses stream events, bounded polling, and kernel socket waits with explicit timeouts.

## Compatibility and roadmap

The legacy object-mode HTTP machinery (the `process.http_servers` table, `net.http_listen`, and the `loopbackHttpTarget` handle) remains only as a compatibility fallback while older callers migrate. It is deprecated and should not grow into a second networking stack; the kernel TCP path is the normal path.

Known follow-ups tracked in the internal networking-bridge spec (`docs/testing/networking-bridge-spec.md`):

- Chunked transfer-encoding decoding in the host fetch response parser. The guest server can emit chunked responses, so the host client needs to decode them.
- Full keep-alive and pipelined request reuse for socket-backed HTTP servers, currently deferred so connections close after draining.
- An optional public host raw-TCP API (`rt.connect({ host, port })`) for cleaner host-to-guest raw TCP, which HTTP dev-server parity does not require.
