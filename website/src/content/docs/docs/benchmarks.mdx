title: "Benchmarks"
description: "Benchmark methodology, scenarios, and results for Secure Exec"
---

These numbers come from the `node-import-bench` Rust binary in the `secure-exec-execution`
crate. It measures how long it takes to start a guest JavaScript isolate and import modules
through the kernel-backed module loader, broken down by phase. The benchmark also runs a set
of host-Node control workloads so isolate overhead can be separated from raw import work.

All "native execution" scenarios run guest code inside a V8 isolate with kernel-backed
polyfills. The "host node" scenarios run the same entrypoints directly under Node.js as a
control, so the difference attributes cost to the isolation layer rather than to the import
work itself.

## What The Benchmark Measures

Each sample is split into four phases:

- **Context setup**: creating the guest JavaScript context.
- **Startup**: bootstrapping the guest runtime (process bootstrap, loader and polyfill wiring)
  before guest module evaluation begins.
- **Guest execution**: the time the guest spends importing and evaluating the workload's
  modules. This is the "import time" the entrypoints self-report.
- **Completion**: draining stdout, collecting results, and tearing the execution down.

"Wall" time is the sum of all four phases. "Startup overhead" is wall time minus guest
execution, that is, everything outside the guest's own import work.

The benchmark runs 5 recorded iterations with 1 warmup iteration discarded. Tail values at
this sample count are indicative, not statistically tight.

## Transport Round Trip

Before the scenarios run, the benchmark measures the round-trip latency of the execution
stdio bridge by echoing payloads of increasing size through a guest process. This gives a
floor to compare the larger startup and import phases against.

| Payload (bytes) | Mean RTT (ms) | p50 (ms) | p95 (ms) |
| --------------- | ------------: | -------: | -------: |
| 32              | 1.07          | 1.07     | 1.09     |
| 4,096           | 1.07          | 1.07     | 1.07     |
| 65,536          | 2.72          | 2.38     | 3.35     |

A small payload round-trips in roughly **1 ms**. The bridge stays flat up to 4 KB and only
grows once the payload reaches 64 KB.

## Startup Floor

These scenarios run a minimal guest entrypoint with no extra imports, so they measure the
pure cost of standing up an isolate.

| Scenario                    | Mean wall (ms) | p50 (ms) | p95 (ms) |
| --------------------------- | -------------: | -------: | -------: |
| `isolate-startup`           | 26.21          | 25.59    | 30.25    |
| `prewarmed-isolate-startup` | 19.30          | 18.96    | 21.13    |

`isolate-startup` is a fresh isolate per sample. `prewarmed-isolate-startup` reuses one
execution engine that keeps materialized assets and polyfill prewarm state alive between
samples, which removes about **7 ms** of repeated bootstrap work.

## Local Module Import

A 24-module local ESM graph that simulates layered application modules. The guest reports its
own import time, and the benchmark separates that from startup overhead.

| Scenario                    | Runtime          | Mean wall (ms) | p50 (ms) | Import (ms) | Startup overhead (ms) |
| --------------------------- | ---------------- | -------------: | -------: | ----------: | --------------------: |
| `cold-local-import`         | native execution | 24.96          | 24.11    | 2.00        | 22.96                 |
| `warm-local-import`         | native execution | 24.97          | 24.89    | 2.00        | 22.97                 |
| `same-context-local-import` | native execution | 21.73          | 21.73    | 2.60        | 19.13                 |
| `prewarmed-local-import`    | native execution | 23.17          | 22.23    | 2.00        | 21.17                 |
| `host-local-import`         | host node        | 21.88          | 21.33    | 2.34        | 19.54                 |

The guest spends only about **2 ms** importing and evaluating the 24-module graph. The rest of
the wall time is isolate startup and result collection, not module work. The host-Node control
(`host-local-import`) lands close to the native path, which shows the isolation layer adds
little over Node's own per-process startup for this workload.

## Builtin Imports

Importing kernel-backed Node builtins through the loader. `builtin-import` is a cold import of
the common bootstrap builtins. The `hot-builtin-*` scenarios are single-import microbenchmarks
that run after a priming pass inside one reused engine, so they isolate the hot import path for
a single builtin.

| Scenario                        | Fixture                                   | Mean wall (ms) | p50 (ms) | Import (ms) |
| ------------------------------- | ----------------------------------------- | -------------: | -------: | ----------: |
| `builtin-import`                | node:path + node:url + node:fs/promises   | 24.98          | 23.96    | 1.00        |
| `hot-builtin-stream-import`     | node:stream                               | 18.80          | 18.71    | 0.40        |
| `hot-builtin-stream-web-import` | node:stream/web                           | 18.73          | 18.19    | 0.00        |
| `hot-builtin-crypto-import`     | node:crypto                               | 18.41          | 18.12    | 0.20        |
| `hot-builtin-zlib-import`       | node:zlib                                 | 19.36          | 19.61    | 0.40        |
| `hot-builtin-url-import`        | node:url                                  | 20.47          | 20.02    | 0.00        |

Once primed, a single builtin import costs well under **1 ms** of guest time. The wall time is
dominated by the isolate startup floor, not by resolving the builtin.

## Real Package Imports

Cold imports of real npm packages from a mounted `node_modules` tree, with representative work
after the import to exercise the package. These run under the host-Node control path so the
import cost reflects the package itself rather than isolate overhead.

| Scenario                            | Fixture                              | Mean wall (ms) | p50 (ms) | Import (ms) |
| ----------------------------------- | ------------------------------------ | -------------: | -------: | ----------: |
| `large-package-import`              | typescript                           | 229.49         | 220.04   | 208.88      |
| `hot-projected-package-file-import` | projected TypeScript compiler file   | 236.55         | 234.13   | 215.28      |
| `projected-package-import`          | projected TypeScript guest-path import | 239.46       | 239.04   | 218.69      |
| `pdf-lib-startup`                   | pdf-lib document creation            | 92.87          | 96.08    | 68.47       |
| `jszip-startup`                     | jszip archive staging                | 33.43          | 30.91    | 15.38       |
| `jszip-end-to-end`                  | jszip end-to-end archive roundtrip   | 40.23          | 40.74    | 23.29       |
| `jszip-repeated-session-compressed` | jszip compressed archive roundtrip   | 47.38          | 48.74    | 27.72       |

Large packages are import-bound, not startup-bound. Importing the `typescript` compiler takes
roughly **209 ms** of guest time, two orders of magnitude more than a builtin import, so the
import path dominates and the startup floor becomes negligible. Smaller packages like `jszip`
import in **15 to 30 ms**, and `pdf-lib` document creation runs in under **100 ms** end to end.

## Test Environment

These figures are environment specific and will vary by hardware. They were measured on:

| Component | Details                                            |
| --------- | -------------------------------------------------- |
| CPU       | 12th Gen Intel Core i7-12700KF (20 logical CPUs)   |
| Node.js   | v24.13.0                                            |
| OS        | Debian GNU/Linux 12 (Linux 6.1 x86_64)             |
| Build     | `--release`                                        |

## Reproducing

```bash
cargo run --release -p secure-exec-execution --bin node-import-bench -- \
  --iterations 5 --warmup-iterations 1
```

The binary writes a Markdown and a JSON report to
`target/benchmark-reports/node-import-bench/` and also prints the Markdown report to stdout.
Pass `--baseline <path>` to compare a run against a previously saved JSON report. Results will
vary by hardware, so treat the numbers above as representative of the test environment rather
than as absolute guarantees.
