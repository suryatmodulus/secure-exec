Secure Exec exists to run code you do not trust: third-party npm packages, code generated by an AI agent, snippets submitted by your users. Instead of hoping that code behaves, Secure Exec runs every program inside a fully virtualized VM where the guest can only ever touch virtual resources, never your real machine. This page is a high-level tour of the protections you get. For the full threat model and the deeper internals, see the [Security Model](/docs/security-model) and the [Architecture](/docs/architecture) reference.

Secure Exec is built on the same technology and security boundary the industry already trusts to isolate untrusted code: guest JavaScript runs inside a **V8 isolate**, the same engine and isolation primitive that powers Google Chrome (Chromium) and Cloudflare Workers. That boundary keeps untrusted web pages and multi-tenant edge code apart in production at enormous scale. Secure Exec runs on that proven foundation and wraps a virtualized POSIX kernel around it, so you also get a virtual filesystem, processes, and network, plus a permission policy and resource limits you control.

## The isolation model

A Secure Exec runtime has three parties:

- **The caller** is your trusted application. It decides what to run and under which policy, then hands the code over for execution.
- **The sidecar** is the enforcement point: the trusted computing base that owns the VM and its kernel. Every guest action is checked here.
- **The executor** runs the guest code and is treated as hostile. It holds no real capabilities of its own.

The isolation boundary is the VM. Guest code only ever interacts with virtualized kernel resources: a virtual filesystem, virtual processes, and a virtual network. It never reaches the real host filesystem, host network, or host processes. Crucially, how code got into the sandbox never makes it trusted. Whether a payload came from a trusted teammate or an anonymous user, it runs behind the same boundary with the same restrictions.

## Secure by default

You do not have to remember to lock things down. A plain `NodeRuntime.create()` boots with a secure baseline policy:

```ts
import { NodeRuntime } from "secure-exec";

// Network is denied by default. Everything stays virtualized.
const rt = await NodeRuntime.create();
```

The network is **denied** out of the box, so guest code cannot reach the internet until you opt in. The virtualized scopes that programs need to run at all (filesystem, child processes, process info, env) are allowed, because they only ever touch the VM, never the host. From there you tighten or widen each scope independently with a simple `"allow"` / `"deny"`, or a rule set for fine-grained control (for example, allow the filesystem but deny one directory, or deny the network except one host).

## Limits that contain abuse

Untrusted code can be slow, wasteful, or actively hostile. The VM keeps it bounded so it can never take down your host:

- A per-run **wall-clock timeout** kills a program once its time budget elapses, so an infinite loop cannot hang you.
- An **`AbortSignal`** lets you cancel an in-flight run from the outside, for example to enforce your own deadline.
- The VM also enforces **memory and CPU limits**, so guest code cannot exhaust host memory or CPU. Runaway loops, memory bombs, and fork bombs are all handled within the VM's bounds.

```ts
// A runaway program is terminated once the budget elapses.
await rt.exec(`while (true) {}`, { timeout: 1000 });
```

## Network confinement

Guest egress is denied by default. When you grant network access, traffic does not open a real host socket: every connection, `fetch()`, and DNS lookup flows through the kernel's virtual socket table, where the permission policy and any allowlist still apply. You can allow the network broadly or restrict it to named destinations.

A server the guest starts is reachable only over loopback inside the VM by default, even when the network is allowed. It stays private unless you explicitly expose its port, so an untrusted program cannot accidentally open a listener to the outside world.

<LinkCard title="Networking" href="/docs/features/networking" description="Guest network access through the kernel socket table, loopback by default." />

## Virtualized identity and environment

The guest does not see your host's identity. It sees a virtualized OS and environment scoped to its own VM: its process identity, its `os.*` values, and its environment variables are all the VM's, not the real machine's. This is per-VM and isolated, so guest code cannot learn about or impersonate the host, and one VM's identity never bleeds into another's.

<LinkCard title="Runtime & Platform" href="/docs/features/runtime-platform" description="The virtualized host environment guest code sees." />

## Filesystem isolation

The guest's filesystem is a virtualized, in-memory filesystem. Reads and writes never reach the real host filesystem. Your host directories are exposed only when you explicitly mount them, and a mount is confined to its root: the guest sees only the subtree you mounted, never the wider host filesystem, and attempts to escape that root (through `..`, symlinks, or path tricks) are rejected. Mounts are read-only unless you opt into writes.

## What this means for you

You can run code you would never run directly on your machine:

- It executes inside a VM with **no path to the real host** filesystem, network, or processes.
- It starts from a **secure default**: the network is off, and you grant only what a given task needs.
- It **cannot exhaust your host**: timeouts, cancellation, and memory and CPU limits keep it bounded.
- It sees a **virtual identity and environment**, not yours, and that state is isolated per VM.
- How the code arrived **never makes it trusted**, so the same protections apply to every payload.

One responsibility stays with you: the host process that creates the runtime is trusted infrastructure. The VM protects your host from the guest, but for internet-facing workloads you should still run your own application in an already-hardened environment. The full in-scope and out-of-scope details are in the [Security Model](/docs/security-model).

<Aside type="caution">Prefer a fresh runtime per untrusted task. Create it, run the task, dispose it, and the next task starts from a clean VM, so an escape attempt cannot outlive a single run.</Aside>
