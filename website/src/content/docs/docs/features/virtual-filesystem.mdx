title: Virtual Filesystem
description: Sandboxed code reads and writes a fully virtual filesystem that lives inside the kernel, isolated from the host disk.

Every VM boots with a **virtual filesystem** that lives entirely inside the kernel. Guest code creates directories, writes files, and lists them with the normal Node `fs` APIs, but none of that I/O ever touches the host disk. The guest sees an ordinary POSIX filesystem; the host sees nothing.

For basic filesystem usage (reading, writing, and listing files from guest code), see [Filesystem & Mounts](/docs/features/filesystem). This page focuses on the VFS model itself: how it isolates the guest, and the mount and storage backends behind it.

## Isolation from the host

The defining property of the virtual filesystem is isolation. A file the guest writes lives only in the kernel's virtual filesystem. When the host checks that same path, it finds nothing: the guest's writes never escape to the host disk. Each VM gets its own filesystem, and disposing the runtime with `rt.dispose()` tears it down.

The example below seeds a file into the VM at boot with `files`, writes another from the host with `rt.writeFile`, has the guest read both, reads one back on the host with `rt.readFile`, and then proves the guest path does not exist on the real host:

```ts
import { NodeRuntime } from "secure-exec";

// Boot a VM and seed a file into its virtual filesystem at boot with `files`.
// The bytes are copied into the kernel's in-memory filesystem; the host
// filesystem is never exposed to the guest.
const rt = await NodeRuntime.create({
	files: { "/home/user/seed.json": JSON.stringify({ ok: true }) },
});

try {
	// Write another file from the host with rt.writeFile, then have the guest
	// read both back. The guest only ever sees the virtual filesystem.
	await rt.writeFile("/home/user/note.txt", "written from the host\n");

	const result = await rt.run(`
		const { readFileSync } = await import("node:fs");
		const seed = JSON.parse(readFileSync("/home/user/seed.json", "utf8"));
		const note = readFileSync("/home/user/note.txt", "utf8").trim();
		console.log("guest read seed:", JSON.stringify(seed));
		console.log("guest read note:", note);
		globalThis.__return({ seed, note });
	`);

	console.log("guest stdout:", result.stdout.trim());
	console.log("guest exit code:", result.exitCode);
	console.log("value returned to host:", result.value);

	// Read a guest-written file back on the host with rt.readFile.
	const bytes = await rt.readFile("/home/user/seed.json");
	console.log("host rt.readFile:", new TextDecoder().decode(bytes));

	// The virtual filesystem is isolated from the host disk. The guest path does
	// not exist on the real host - prove it by checking the same path here.
	const { existsSync } = await import("node:fs");
	const guestPath = "/home/user/seed.json";
	console.log(
		`host sees ${guestPath}?`,
		existsSync(guestPath) ? "YES (unexpected!)" : "NO - isolated from host",
	);
} finally {
	await rt.dispose();
}
```

Running this prints:

```text
guest stdout: guest read seed: {"ok":true}
guest read note: written from the host
guest exit code: 0
value returned to host: { seed: { ok: true }, note: 'written from the host' }
host rt.readFile: {"ok":true}
host sees /home/user/seed.json? NO - isolated from host
```

This is the core guarantee of the virtual filesystem: guest code runs against a faithful POSIX filesystem, but every read and write is mediated by the kernel and stays inside the isolation boundary.

## Seeding and exchanging files

`NodeRuntime` gives you three ways to move data across the isolation boundary without exposing the host disk:

- **`create({ files })`** seeds files into the virtual filesystem before the guest runs, keyed by absolute guest path. Parent directories are created as needed. Use it to project fixtures, config, or npm packages into the sandbox. The bytes are copied into the VM's in-memory filesystem.
- **`rt.writeFile(path, data)`** writes a file into the VM after boot, creating parent directories as needed.
- **`rt.readFile(path)`** reads a file from the VM back to the host as raw bytes.

All three operate on the virtual filesystem only; the host filesystem is never touched.

## Custom and host-backed filesystems

The default `NodeRuntime.create()` filesystem is purely in-memory: it starts empty (apart from the bootstrapped root) and lives only for the life of the VM. That is not the only backing the kernel supports. Secure Exec mounts are pluggable, and several backends are real today.

- **Host directory mounts.** Project a real host directory into the VM with `NodeRuntime.create({ mounts })`. This is a first-class `NodeRuntime` option: see [Mounting host directories](/docs/features/filesystem#mounting-host-directories) for a runnable example. The mount is read-only by default and exposes only the chosen subtree to the guest.
- **Overlay filesystems.** The kernel can layer a writable filesystem over a read-only base, so guest writes land in the overlay and the base is never mutated. Overlays are configured at the mount/protocol layer rather than through a `NodeRuntime.create()` option.
- **Pluggable storage backends.** Remote and cloud-backed filesystems ship as separate packages that build a native mount plugin descriptor:
  - **`@secure-exec/s3`** (`createS3Backend`) backs a mount with an Amazon S3 bucket, so guest code reads and writes the bucket as an ordinary POSIX directory.
  - **`@secure-exec/google-drive`** (`createGoogleDriveBackend`) backs a mount with Google Drive, exposing Drive files to the guest as a POSIX directory.
  - **`@secure-exec/sandbox`** (`createSandboxFs`) backs a mount with a remote sandbox-agent host, scoping all guest filesystem operations under a base path on that host.

Host directory mounts are available directly through `NodeRuntime.create({ mounts })`. The overlay and remote storage backends are wired at the lower-level mount and descriptor layer: each storage package returns a mount plugin descriptor that is attached through the VM configuration (`configureVm` mounts) rather than through a one-line `create()` flag. They are real Secure Exec capabilities, driven through the protocol-level mount plumbing.

## Notes

- Guest code runs as an ES module. When you pass code to `rt.run(...)`, it is wrapped so that `globalThis.__return(...)` can hand a value back to the host. Because of that wrapping, use dynamic `await import("node:fs")` inside `run` code rather than a top-level `import` statement. (`rt.exec(...)`, which only captures stdout/stderr, runs your code as a plain module and accepts top-level `import`.)
- The default working directory is `/home/user`. Pass `cwd` to `NodeRuntime.create({ cwd })` to change it.
- File access is governed by the VM permission policy. By default the runtime allows the filesystem scope; tighten it by passing `permissions` to `NodeRuntime.create(...)`.
