Every VM boots with a **virtual filesystem (VFS)** that lives entirely inside the kernel. Guest code uses the normal Node `fs` APIs, but none of that I/O touches the host disk. The guest sees an ordinary POSIX filesystem; the host sees nothing.

This page covers the VFS model: isolation, the data-exchange APIs that cross the boundary, and the mount and storage backends behind it. For day-to-day reading, writing, and listing from guest code, see [Filesystem & Mounts](/docs/features/filesystem).

## The model

- Each VM gets its own in-memory VFS, bootstrapped from a root filesystem snapshot.
- Guest `fs` calls are mediated by the kernel and stay inside the isolation boundary. Nothing escapes to the host disk.
- The default VFS is purely in-memory and lives only for the life of the VM. `rt.dispose()` tears it down.
- Mounts are pluggable: a guest path can be backed by the host, a remote, or a cloud store (see below).

## Isolation from the host

A file the guest writes lives only in the kernel VFS. When the host checks that same path, it finds nothing. The example below seeds a file at boot with `files`, writes another from the host with `rt.writeFile`, has the guest read both, reads one back on the host with `rt.readFile`, then proves the guest path does not exist on the real host:

```ts
import { NodeRuntime } from "secure-exec";

// Seed a file into the VFS at boot. Bytes are copied into the kernel's
// in-memory filesystem; the host filesystem is never exposed to the guest.
const rt = await NodeRuntime.create({
	files: { "/home/user/seed.json": JSON.stringify({ ok: true }) },
});

try {
	// Write another file from the host after boot.
	await rt.writeFile("/home/user/note.txt", "written from the host\n");

	const result = await rt.run(`
		const { readFileSync } = await import("node:fs");
		const seed = JSON.parse(readFileSync("/home/user/seed.json", "utf8"));
		const note = readFileSync("/home/user/note.txt", "utf8").trim();
		console.log("guest read seed:", JSON.stringify(seed));
		console.log("guest read note:", note);
		globalThis.__return({ seed, note });
	`);

	console.log("guest stdout:", result.stdout.trim());
	console.log("guest exit code:", result.exitCode);
	console.log("value returned to host:", result.value);

	// Read a guest-written file back on the host.
	const bytes = await rt.readFile("/home/user/seed.json");
	console.log("host rt.readFile:", new TextDecoder().decode(bytes));

	// The guest path does not exist on the real host. Prove it.
	const { existsSync } = await import("node:fs");
	const guestPath = "/home/user/seed.json";
	console.log(
		`host sees ${guestPath}?`,
		existsSync(guestPath) ? "YES (unexpected!)" : "NO - isolated from host",
	);
} finally {
	await rt.dispose();
}
```

Prints:

```text
guest stdout: guest read seed: {"ok":true}
guest read note: written from the host
guest exit code: 0
value returned to host: { seed: { ok: true }, note: 'written from the host' }
host rt.readFile: {"ok":true}
host sees /home/user/seed.json? NO - isolated from host
```

## Moving data across the boundary

Three `NodeRuntime` APIs cross the isolation boundary without exposing the host disk. All operate on the VFS only.

```ts
// Seed files before the guest runs, keyed by absolute guest path.
// Parent dirs are created. Values are string | Uint8Array.
const rt = await NodeRuntime.create({
	files: { "/home/user/config.json": '{"ok":true}' },
});

// Write a file after boot (creates parent dirs).
await rt.writeFile("/home/user/out.txt", "hi");          // string | Uint8Array

// Read a file back to the host as raw bytes.
const bytes = await rt.readFile("/home/user/out.txt");   // Promise<Uint8Array>
```

- **`create({ files })`**: seed fixtures, config, or npm packages into the VFS before boot. Bytes are copied into the in-memory filesystem.
- **`rt.writeFile(path, data)`**: write into the VFS after boot.
- **`rt.readFile(path)`**: read from the VFS as `Uint8Array`. Decode with `new TextDecoder().decode(bytes)` for text.

Guest programs that need stdin or live output use the exec options instead: `rt.exec`/`rt.run` accept `{ env, cwd, stdin, timeout, signal, onStdout, onStderr }`, and `rt.spawn` returns a live handle for long-running guests. See [Child Processes](/docs/features/child-processes) and [Output Capture](/docs/features/output-capture) for those.

## Host directory mounts

Project a real host directory into the VFS, Docker-style, with `create({ mounts })`. This is the one host-backed backend wired directly into `NodeRuntime`.

```ts
const rt = await NodeRuntime.create({
	mounts: [
		{
			guestPath: "/home/user/project/node_modules/typescript",
			hostPath: "/abs/path/to/node_modules/typescript",
			readOnly: true, // default; pass false to allow guest writes
		},
	],
});
```

Each mount is a `HostDirectoryMount`:

- **`guestPath`**: absolute guest path the directory appears at.
- **`hostPath`**: absolute host directory to project.
- **`readOnly`**: read-only by default; `false` allows guest writes.

Files are read **lazily** through the VFS as the guest accesses them, so large trees (a `node_modules` package, a repo checkout) project without copying their bytes up front. The guest sees only the mounted subtree, never the wider host filesystem. For a runnable example see [Mounting host directories](/docs/features/filesystem#mounting-host-directories).

## Remote and cloud backends

Beyond host mounts, a guest path can be backed by a remote or cloud store. Each backend ships as its own package and returns a mount descriptor that is attached at the lower-level VM mount layer (not through a `create()` flag). Guest code reads and writes the mount as an ordinary POSIX directory.

- **`@secure-exec/s3`** (`createS3Backend`): backs a mount with an Amazon S3 bucket.
- **`@secure-exec/google-drive`** (`createGoogleDriveBackend`): backs a mount with Google Drive.
- **`@secure-exec/sandbox`** (`createSandboxFs`): backs a mount with a remote sandbox-agent host, scoping guest filesystem operations under a base path.

These are real Secure Exec capabilities driven through the protocol-level mount plumbing. The kernel can also layer a writable overlay over a read-only base so guest writes land in the overlay and the base is never mutated; overlays are configured at the mount layer rather than through a `create()` option.

## Notes

- Guest code runs as an ES module. `rt.run(...)` injects `globalThis.__return(...)` so the guest can hand a value back to the host; use dynamic `await import("node:fs")` inside `run` code (top-level `import` also works, since the preamble is hoisted). `rt.exec(...)` captures only stdout/stderr.
- The default working directory is `/home/user`. Override with `create({ cwd })`.
- File access is governed by the VM permission policy. The default grants the filesystem scope; tighten it with `create({ permissions })`.
