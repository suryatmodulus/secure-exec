Every guest syscall is checked against a per-scope permission policy passed as `permissions` to `NodeRuntime.create()`.

- Six scopes, configured independently: `fs`, `network`, `childProcess`, `process`, `env`, `tool`.
- Each scope is `"allow"`, `"deny"`, or a rule set.
- A denied operation is rejected with `EACCES` before any host resource is touched.
- The policy you pass is **merged over a secure default**, so partial policies work.

## Defaults and merge semantics

Secure Exec is deny-by-default for outward-facing capabilities. When you pass no `permissions`, this baseline applies:

```ts
{
  fs: "allow",          // virtualized in-memory filesystem only
  childProcess: "allow",
  process: "allow",
  env: "allow",
  network: "deny",      // no network egress until you opt in
}
```

- `fs`/`childProcess`/`process`/`env` are allowed because they are fully virtualized (the guest sees only the VM, never the host) and are required to run a program at all.
- `network` is denied: guest code cannot reach the network until you opt in.
- Your `permissions` are merged **over** this baseline. Omitted scopes keep their default; they are **not** denied. So `{ network: "allow" }` grants the network while keeping the execution essentials.

```ts
// Grant the network, leave everything else at the secure default.
const rt = await NodeRuntime.create({ permissions: { network: "allow" } });
```

## Configuring Permissions

The example below allows the filesystem but denies the network. The guest writes and reads a file successfully, while an outbound `fetch()` is blocked by the kernel.

```ts
import { NodeRuntime } from "secure-exec";

// Permissions are a per-scope policy evaluated against every guest syscall.
// Each scope (fs, network, childProcess, process, env, tool) is configured
// independently as "allow", "deny", or a rule set. Here the guest may use the
// filesystem, but the network is fully denied - so an outbound fetch is blocked
// by the kernel before any socket is opened.
const rt = await NodeRuntime.create({
  permissions: {
    fs: "allow",
    network: "deny",
    childProcess: "allow",
    process: "allow",
    env: "allow",
  },
});

try {
  // Guest code runs as an ES module inside the VM. The filesystem write/read
  // succeeds because fs is allowed; the fetch() throws because the kernel
  // rejects the connection with EACCES from the network policy.
  const { value, stdout, stderr, exitCode } = await rt.run<{
    fileContents: string;
    networkBlocked: boolean;
    networkError: string | null;
  }>(`
    const { mkdirSync, writeFileSync, readFileSync } = await import("node:fs");

    // Allowed: filesystem access is permitted.
    mkdirSync("/workspace", { recursive: true });
    writeFileSync("/workspace/note.txt", "written inside the sandbox");
    const fileContents = readFileSync("/workspace/note.txt", "utf8");
    console.log("filesystem allowed:", fileContents);

    // Denied: the network scope is set to "deny", so fetch() is blocked.
    let networkBlocked = false;
    let networkError = null;
    try {
      await fetch("http://example.com");
      console.log("UNEXPECTED: fetch succeeded");
    } catch (error) {
      networkBlocked = true;
      networkError = (error.cause && error.cause.message) || error.message;
      console.log("network denied:", networkError);
    }

    __return({ fileContents, networkBlocked, networkError });
  `);

  console.log("---");
  console.log("exitCode:", exitCode);
  if (stderr.trim()) console.log("guest stderr:", stderr.trim());
  console.log("guest stdout:\n" + stdout.trim());
  console.log("file contents:", value?.fileContents);
  console.log("network blocked:", value?.networkBlocked);
  console.log("network error:", value?.networkError);

  if (value?.fileContents !== "written inside the sandbox") {
    throw new Error("expected allowed filesystem write/read to succeed");
  }
  if (!value?.networkBlocked) {
    throw new Error("expected denied network fetch to be blocked");
  }
  console.log("OK: filesystem allowed, network denied");
} finally {
  await rt.dispose();
}
```

## Permission scopes

| Scope | Controls | Default |
|---|---|---|
| `fs` | Filesystem reads, writes, and metadata operations | `allow` |
| `network` | Outbound connections, `fetch`, HTTP, and DNS | `deny` |
| `childProcess` | Spawning child processes | `allow` |
| `process` | Process-control operations | `allow` |
| `env` | Environment variable access | `allow` |
| `tool` | Invoking host tools registered via `tools` / `rt.registerTools` | `deny`* |

\* The `tool` scope is auto-granted to `allow` when you pass `tools` to `create()` and set no `tool` policy of your own. Pass `permissions.tool` to gate individual tools.

## Allow, deny, and rule sets

The simplest value for a scope is `"allow"` or `"deny"`:

```ts
const rt = await NodeRuntime.create({
  permissions: { fs: "allow", network: "deny" },
});
```

For finer control, a scope can be a rule set: a `default` mode plus an ordered list of `rules` evaluated in order.

- `default`: `"allow"` or `"deny"`, applied when no rule matches.
- Each rule has a `mode` (`"allow"` or `"deny"`) and matchers.
- `fs` rules match on `operations` and `paths` (path/glob patterns).
- All other scopes match on `operations` and `patterns`.
- Use `["*"]` as a wildcard for any operation or pattern.

```ts
// fs: allow by default, but deny anything under /vault.
const rt = await NodeRuntime.create({
  permissions: {
    fs: {
      default: "allow",
      rules: [{ mode: "deny", operations: ["*"], paths: ["/vault/**"] }],
    },
  },
});
```

```ts
// network: deny by default, allow only api.example.com.
const rt = await NodeRuntime.create({
  permissions: {
    network: {
      default: "deny",
      rules: [{ mode: "allow", operations: ["*"], patterns: ["api.example.com"] }],
    },
  },
});
```

## Tools and the `tool` scope

Host tools registered with `tools` (at `create`) or `rt.registerTools` (after boot) are gated by the `tool` scope. The `create` option auto-grants `tool: "allow"` when you set no policy; `rt.registerTools` does not, so grant it yourself:

```ts
const rt = await NodeRuntime.create({
  permissions: { tool: "allow" },
});
await rt.registerTools({
  add: {
    description: "Add two numbers",
    handler: ({ a, b }: { a: number; b: number }) => ({ sum: a + b }),
  },
});
```

Restrict which tools the guest may call with a rule set over their command names:

```ts
const rt = await NodeRuntime.create({
  tools: { add: { /* ... */ }, deleteFile: { /* ... */ } },
  permissions: {
    tool: { default: "deny", rules: [{ mode: "allow", patterns: ["add"] }] },
  },
});
```
