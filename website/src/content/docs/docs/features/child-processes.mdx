Sandboxed code can spawn child processes with the standard `node:child_process` module. Each child runs as a kernel-managed process inside the VM. It never spawns a real host process or escapes the isolation boundary.

## Spawning Child Processes

The guest spawns commands available in the VM (`sh` and `node`) and reads their output:

```ts
import { NodeRuntime } from "secure-exec";

// Boot a fully virtualized VM. The guest runs inside the kernel isolation
// boundary, and any child processes it spawns are themselves kernel-managed
// processes - they never spawn real host processes.
const rt = await NodeRuntime.create();

try {
  // Guest code runs as an ES module inside the VM. It uses the standard
  // node:child_process module to spawn a command available in the VM and read
  // its output. Here we spawn `sh -c` to echo a message, and `node` to report
  // its own version - both run as kernel-managed child processes.
  const { stdout, stderr, exitCode } = await rt.exec(`
    import { execFileSync } from "node:child_process";

    // Spawn a shell command and capture its stdout.
    const shellOut = execFileSync("sh", ["-c", "echo hello from a child process"], {
      encoding: "utf8",
    });
    console.log("sh output:", shellOut.trim());

    // Spawn node as a child process and read its version.
    const nodeVersion = execFileSync("node", ["--version"], {
      encoding: "utf8",
    });
    console.log("child node version:", nodeVersion.trim());
  `);

  console.log("exitCode:", exitCode);
  if (stderr.trim()) console.log("guest stderr:", stderr.trim());
  console.log("guest stdout:");
  console.log(stdout.trim());
} finally {
  await rt.dispose();
}
```

*[See Full Example](https://github.com/rivet-dev/secure-exec/tree/main/examples/docs/feat-child-processes)*

Running this prints the output captured from each child process:

```
exitCode: 0
guest stdout:
sh output: hello from a child process
child node version: v22.0.0
```

## Available commands

Children can run any command provided by the runtimes mounted in the VM. By default the VM ships a WASM-backed `sh` and coreutils plus a V8-backed `node`, so `execFileSync("node", ["--version"])` and `execFileSync("sh", ["-c", "…"])` both work out of the box.

You can point the VM at a different set of WASM command binaries by passing `commandsDir` to `NodeRuntime.create()`:

```ts
const rt = await NodeRuntime.create({
  commandsDir: "/path/to/wasm/commands",
});
```

## Other child_process APIs

The example uses `execFileSync` for brevity, but the asynchronous and streaming `node:child_process` APIs (`spawn`, `exec`, `execFile`) also work. Use them when you need to stream stdout/stderr incrementally or write to a child's stdin while it runs.

<Aside type="note">Child processes always run inside the kernel. There is no way for guest code to reach a real host process or host binary, since `node:child_process` only sees the commands provided by the VM's mounted runtimes.</Aside>
