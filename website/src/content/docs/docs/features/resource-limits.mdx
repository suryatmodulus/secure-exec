Resource limits prevent sandboxed code from running forever or exhausting the
host. Each `exec()` call accepts a `timeout`, and when the budget elapses the
guest process is killed so a runaway program can never hang the host. The
timeout is the limit you reach for through the ergonomic `NodeRuntime` API
today; secure-exec enforces several more resource limits at the VM level, and
those are described at the end of this page.

## Limiting Execution Time

```ts
import { NodeRuntime } from "secure-exec";

const rt = await NodeRuntime.create();

try {
	// A normal program finishes well within the timeout budget.
	const ok = await rt.exec(`console.log("finished work");`, {
		timeout: 5000,
	});
	console.log("normal run:");
	console.log("  exitCode:", ok.exitCode);
	console.log("  stdout:", JSON.stringify(ok.stdout.trim()));

	// A runaway program (infinite loop) never returns on its own. The exec
	// timeout terminates the guest process after the budget elapses.
	const start = Date.now();
	const runaway = await rt.exec(`while (true) {}`, {
		timeout: 1000,
	});
	const elapsed = Date.now() - start;

	console.log("runaway run (timeout: 1000ms):");
	console.log("  exitCode:", runaway.exitCode);
	console.log("  elapsedMs:", elapsed);

	// A killed process exits non-zero; a clean exit would be 0.
	const terminated = runaway.exitCode !== 0;
	console.log(
		terminated
			? "runaway guest was terminated by the timeout"
			: "ERROR: runaway guest was NOT terminated",
	);

	if (!terminated) {
		process.exitCode = 1;
	}
} finally {
	await rt.dispose();
}
```

Running it prints:

```text
normal run:
  exitCode: 0
  stdout: "finished work"
runaway run (timeout: 1000ms):
  exitCode: 137
  elapsedMs: 2506
runaway guest was terminated by the timeout
```

## Execution timeout

Pass `timeout` (in milliseconds) to `exec()` to bound how long a single guest
program may run. When the budget elapses, the guest process is killed and `exec()`
returns with a non-zero `exitCode` (`137`, i.e. `128 + SIGKILL`).

```ts
const runaway = await rt.exec(`while (true) {}`, {
	timeout: 1000,
});

console.log(runaway.exitCode); // 137 (terminated)
```

A program that finishes before the timeout returns its real exit code:

```ts
const ok = await rt.exec(`console.log("finished work");`, {
	timeout: 5000,
});

console.log(ok.exitCode); // 0
```

The same `timeout` option is available on `run()`.

## Other limits enforced by the VM

The execution `timeout` is the only resource limit exposed as a
`NodeRuntime.exec()` / `NodeRuntime.run()` option today, but it is not the only
limit secure-exec enforces. The VM applies a broader set of resource limits
configured at VM creation through the protocol's `VmLimits`
(`CreateVmConfig.limits`), including:

- **Memory**: a V8 heap limit (`jsRuntime.v8HeapLimitMb`) caps how much memory
  guest JavaScript may allocate, plus filesystem byte and inode caps for the
  virtual filesystem.
- **Payload and transfer limits**: caps on captured output, stdin buffering,
  event payload size, and `fetch()` response size bound how much data moves
  between the guest and the host.
- **CPU-time budget**: secure-exec can bound CPU usage (cores and blocking
  read time) independently of the wall-clock execution `timeout`.
- **Timing mitigation**: secure-exec can freeze timers to blunt timing side
  channels.

These limits are real and enforced by secure-exec, but they are not yet
surfaced as `NodeRuntime.create()` options. The ergonomic `NodeRuntime` API
boots the VM with secure-exec's defaults for them; only the execution
`timeout` is configurable per call today. To tune the other limits you set
`CreateVmConfig.limits` when driving the VM through the lower-level protocol.
