Bound and cancel guest execution so runaway code can never hang the host:

- `timeout` (ms) on `exec()` / `run()`: kill the guest after a wall-clock budget.
- `signal` (`AbortSignal`) on `exec()` / `run()`: cancel an in-flight run from the outside.

Memory, payload, CPU-time, and timing limits are also enforced by the VM, but are not yet `NodeRuntime` options (see [Other limits](#other-limits-enforced-by-the-vm)).

## Limiting execution time

```ts
import { NodeRuntime } from "secure-exec";

const rt = await NodeRuntime.create();

try {
	// A normal program finishes well within the timeout budget.
	const ok = await rt.exec(`console.log("finished work");`, {
		timeout: 5000,
	});
	console.log("normal run:");
	console.log("  exitCode:", ok.exitCode);
	console.log("  stdout:", JSON.stringify(ok.stdout.trim()));

	// A runaway program (infinite loop) never returns on its own. The exec
	// timeout terminates the guest process after the budget elapses.
	const start = Date.now();
	const runaway = await rt.exec(`while (true) {}`, {
		timeout: 1000,
	});
	const elapsed = Date.now() - start;

	console.log("runaway run (timeout: 1000ms):");
	console.log("  exitCode:", runaway.exitCode);
	console.log("  elapsedMs:", elapsed);

	// A killed process exits non-zero; a clean exit would be 0.
	const terminated = runaway.exitCode !== 0;
	console.log(
		terminated
			? "runaway guest was terminated by the timeout"
			: "ERROR: runaway guest was NOT terminated",
	);

	if (!terminated) {
		process.exitCode = 1;
	}
} finally {
	await rt.dispose();
}
```

Running it prints:

```text
normal run:
  exitCode: 0
  stdout: "finished work"
runaway run (timeout: 1000ms):
  exitCode: 137
  elapsedMs: 2506
runaway guest was terminated by the timeout
```

## `timeout`

- Pass `timeout` in milliseconds to bound how long a single guest program may run.
- When the budget elapses the guest process is killed; the call **resolves** with a non-zero `exitCode` (`137`, i.e. `128 + SIGKILL`).
- A program that finishes first resolves with its real exit code.

```ts
const runaway = await rt.exec(`while (true) {}`, { timeout: 1000 });
console.log(runaway.exitCode); // 137 (terminated)

const ok = await rt.exec(`console.log("done");`, { timeout: 5000 });
console.log(ok.exitCode); // 0
```

`timeout` is available on both `exec()` and `run()`.

## `signal` (AbortSignal)

- Pass an `AbortSignal` to cancel an in-flight run from the outside.
- On abort the guest process is killed inside the VM (the kernel delivers `SIGTERM`) and the call **rejects** with the signal's abort reason (an `AbortError` by default).
- Aborting an already-finished run has no effect; passing an already-aborted signal rejects immediately.

```ts
const controller = new AbortController();
const pending = rt.exec(`while (true) {}`, { signal: controller.signal });

controller.abort();

await pending.catch((err) => console.log(err.name)); // "AbortError"
```

Wire a signal to your own deadline, or cancel work when an upstream request is canceled:

```ts
// Cancel after 2s, surfacing your own reason on the rejection.
const result = await rt.exec(code, {
	signal: AbortSignal.timeout(2000),
});
```

`timeout` and `signal` can be combined: whichever fires first ends the run (the `timeout` kills and resolves non-zero, the `signal` kills and rejects).

## Result shape

`exec()` resolves with:

```ts
interface NodeRuntimeExecResult {
	stdout: string;
	stderr: string;
	exitCode: number;
}
```

`run()` resolves with the same fields plus `value`, the JSON value the guest returned via `__return(...)` (only when `exitCode === 0`):

```ts
interface NodeRuntimeRunResult<T> {
	value?: T;
	stdout: string;
	stderr: string;
	exitCode: number;
}
```

## Full option set for `exec()` / `run()`

Both take the same options:

- `timeout?: number`: kill the run after this many milliseconds (resolves non-zero).
- `signal?: AbortSignal`: cancel the run; rejects with the abort reason.
- `stdin?: string | Uint8Array`: data piped to the guest's stdin.
- `env?: Record<string, string>`: extra env merged over the VM env for this run.
- `cwd?: string`: working directory for this run.
- `onStdout?: (chunk: Uint8Array) => void`: stream stdout chunks as produced.
- `onStderr?: (chunk: Uint8Array) => void`: stream stderr chunks as produced.

```ts
const out = await rt.exec(code, {
	timeout: 5000,
	stdin: "hello\n",
	env: { LOG: "1" },
	onStdout: (chunk) => process.stdout.write(chunk),
});
```

For long-running guests that should not run to completion (dev servers, watchers), use `rt.spawn(code, options)` instead, which returns a live `NodeRuntimeProcess` you `kill()` and `wait()` on directly.

## Other limits enforced by the VM

`timeout` and `signal` are the only execution-bounding controls exposed as `NodeRuntime` options today. The VM enforces a broader set of limits, configured at VM creation through the protocol's `VmLimits` (`CreateVmConfig.limits`):

- **Memory**: a V8 heap limit (`jsRuntime.v8HeapLimitMb`) caps guest JavaScript allocation, plus filesystem byte and inode caps for the virtual filesystem.
- **Payload and transfer**: caps on captured output, stdin buffering, event payload size, and `fetch()` response size bound how much data moves between guest and host.
- **CPU-time budget**: bounds CPU usage (cores and blocking read time) independently of the wall-clock `timeout`.
- **Timing mitigation**: timers can be frozen to blunt timing side channels.

These limits are real and enforced, but not yet surfaced as `NodeRuntime.create()` options. The ergonomic API boots the VM with Secure Exec's defaults for them; only `timeout`/`signal` are configurable per call. To tune the rest, set `CreateVmConfig.limits` when driving the VM through the lower-level protocol.
