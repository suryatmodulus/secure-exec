`@secure-exec/typescript` runs the TypeScript compiler **inside the sandbox**. The `typescript` package is projected into the VM's virtual filesystem and every `createProgram`/`emit` call happens in the guest, so untrusted TypeScript never compiles or runs on the host.

**Why compile in the sandbox:** type-checking attacker-controlled source is a **CPU and memory amplification** vector. A small malicious type (recursive conditional types, deeply nested generics) can make `tsc` spin for seconds or balloon its heap. Because the compiler runs in a disposable VM, the blow-up is contained by the sandbox boundary, and you can bound each run with a `timeout` instead of hanging or OOM-ing the host.

## Install

```bash
npm install @secure-exec/typescript secure-exec
```

## Compile and type-check a source string

```ts
import { createTypeScriptTools } from "@secure-exec/typescript";
import { NodeRuntime } from "secure-exec";

const tools = createTypeScriptTools();

const typeScriptSource = `
interface Greeting {
  name: string;
  count: number;
}

const greeting: Greeting = { name: "secure-exec", count: 3 };
const lines: string[] = Array.from(
  { length: greeting.count },
  (_unused, index) => \`hello \${greeting.name} #\${index + 1}\`,
);

for (const line of lines) {
  console.log(line);
}
`;

// Step 1: compile TypeScript to JavaScript inside the sandbox.
const compiled = await tools.compileSource({
  sourceText: typeScriptSource,
  compilerOptions: { module: "ESNext", target: "ES2022" },
});

if (!compiled.success) {
  const messages = compiled.diagnostics.map((d) => d.message);
  throw new Error(`TypeScript compile failed:\n${messages.join("\n")}`);
}

console.log("Compiled TypeScript to JavaScript inside the sandbox.");

// Step 2: run the emitted JavaScript inside the sandbox.
const rt = await NodeRuntime.create();
try {
  const result = await rt.exec(compiled.outputText ?? "");
  console.log("exitCode:", result.exitCode);
  console.log("guest stdout:\n" + result.stdout.trimEnd());
} finally {
  await rt.dispose();
}

// Step 3: type-check a snippet that has a type error inside the sandbox.
const typeCheck = await tools.typecheckSource({
  sourceText: `const total: number = "not a number";`,
});

console.log("type check success:", typeCheck.success);
for (const diagnostic of typeCheck.diagnostics) {
  console.log(
    `  ${diagnostic.category} TS${diagnostic.code} (line ${diagnostic.line}): ${diagnostic.message}`,
  );
}

if (typeCheck.success) {
  throw new Error("Expected the ill-typed snippet to fail type checking.");
}

console.log("OK: TypeScript compiled and type-checked inside the sandbox.");
```

## `createTypeScriptTools(options?)`

Returns the four in-sandbox tools below. Each call boots a fresh VM, runs the compiler in the guest, and disposes the VM. Options (all optional):

- `compilerPackageDir`: host directory of the `typescript` package to project. Defaults to the `typescript` resolved from this package.
- `compilerGuestDir`: guest mount path for the compiler. Defaults to `/root/node_modules/typescript`.
- `files: Record<string, string | Uint8Array>`: files to seed into the VM (e.g. a `tsconfig.json` or sources), keyed by absolute guest path.
- `mounts: HostDirectoryMount[]`: host directories to project into the VM, Docker-style (lazy reads).
- `permissions`: VM permission policy (same shape as `NodeRuntime.create`).
- `env: Record<string, string>`: environment variables for the guest compiler.

## Tools

```ts
const tools = createTypeScriptTools();

// Single source string
await tools.compileSource({ sourceText, compilerOptions?, filePath?, cwd?, configFilePath? });
await tools.typecheckSource({ sourceText, ... });

// tsconfig.json project (uses files/mounts seeded above)
await tools.compileProject({ cwd?, configFilePath? });
await tools.typecheckProject({ cwd?, configFilePath? });
```

- **`compileSource(options)`** → `{ success, diagnostics, outputText?, sourceMapText? }`. `outputText` is the emitted JavaScript, ready to run with `NodeRuntime`.
- **`typecheckSource(options)`** → `{ success, diagnostics }`. Type-checks without emitting.
- **`compileProject(options?)`** → `{ success, diagnostics, emitSkipped, emittedFiles }`. Emits into the VM filesystem.
- **`typecheckProject(options?)`** → `{ success, diagnostics }`.

`SourceCompilerOptions`:

- `sourceText` (required): the TypeScript to compile or check.
- `filePath?`: virtual path the source appears at. Defaults to a temp `.ts` file.
- `cwd?`: working directory inside the VM. Defaults to `/root`.
- `configFilePath?`: a `tsconfig.json` whose `compilerOptions` are applied.
- `compilerOptions?`: inline compiler options in tsc JSON spelling. Defaults are `target: ES2022`, `module: ESNext`.

`ProjectCompilerOptions`: `cwd?` (defaults to `/root`) and `configFilePath?` (defaults to the `tsconfig.json` found from `cwd`).

Every diagnostic is structured: `{ code, category, message, filePath?, line?, column? }`, where `category` is `"error" | "warning" | "suggestion" | "message"`. `success` is `false` when any diagnostic is an error.

## Compiling a project

Seed a `tsconfig.json` and sources with `files`, or project a host directory with `mounts`:

```ts
const tools = createTypeScriptTools({
  files: {
    "/root/tsconfig.json": JSON.stringify({
      compilerOptions: { strict: true, target: "ES2022", module: "ESNext" },
      include: ["src"],
    }),
    "/root/src/index.ts": "export const answer: number = 42;\n",
  },
});

const result = await tools.compileProject({ cwd: "/root" });
// result.emittedFiles -> paths written into the VM filesystem
```

`compileProject` emits into the VM's virtual filesystem. To pull emitted output back to the host, run the same compile manually on a `NodeRuntime` and read the files with `rt.readFile`.

## Bounding compiler runs

`compileSource`/`typecheckSource` manage their own VM, so to bound an *untrusted* compile yourself, run the emitted output (or your own compiler driver) on a `NodeRuntime` and pass `exec`/`run` options:

```ts
const rt = await NodeRuntime.create();
try {
  // Bound the run: abort after 5s, or via an AbortSignal.
  const out = await rt.exec(compiled.outputText ?? "", {
    timeout: 5_000,
    signal: AbortSignal.timeout(10_000),
    onStdout: (chunk) => process.stdout.write(chunk),
  });
  console.log(out.exitCode);
} finally {
  await rt.dispose();
}
```

`exec`/`run` options: `{ env, cwd, stdin, timeout, signal, onStdout, onStderr }`. On `timeout` or `signal` abort the guest process is killed inside the VM (`SIGTERM`), so a runaway compile can never hang the host.
