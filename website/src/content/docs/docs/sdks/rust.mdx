The Rust client lives in the `secure-exec-client` crate. It speaks the same sidecar wire protocol as the TypeScript client, so every capability reachable from `@secure-exec/core` is also reachable from Rust.

<Aside type="note">
The Rust API reference is generated with rustdoc, not maintained by hand. Use the generated docs as the source of truth for signatures and types.
</Aside>

## What the crate exposes

The entry point is `SidecarProcess`, the low-level handle to a running sidecar (the analogue of the TypeScript `SidecarProcess`).

```rust
use secure_exec_client::SidecarProcess;

// Spawn a sidecar (pass an explicit binary path, or None to resolve it).
let sidecar = SidecarProcess::spawn(None).await?;
```

Key items re-exported from the crate root:

| Item | Description |
|---|---|
| `SidecarProcess` | Spawns and owns a sidecar process; sends wire requests and subscribes to events. |
| `wire::*` | The wire protocol types (shared with `secure-exec-sidecar`), including `CreateVmConfig`, request/response payloads, and events. |
| `TransportError`, `TransportResult`, `ProtocolCodecError` | Transport and codec error types. |
| `WireSidecarCallback` | Host callback registration for sidecar-to-host calls. |

You drive a VM by sending wire requests through `SidecarProcess` (create a VM with `CreateVmConfig`, execute guest code, stream events). The wire types are identical to those the TypeScript client uses, so the [TypeScript SDK](/docs/sdks/typescript) reference is a useful companion for the per-capability shapes.

<Aside type="caution">
The sidecar wire protocol is same-version lockstep: the client and sidecar must be built from the same version. There is no protocol versioning or negotiation.
</Aside>
